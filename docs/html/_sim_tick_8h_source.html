116 /** This class define the interface to provide a simulates system tick (in
117  milliseconds) to the application. The goal is provide a mechanism for
118  an application to run in 'simulate time' instead of real time. The
119  simulated tick mechanism is for the most part transparent an application,
120  i.e. the application programmer does NOT have to be aware or design
121  his code differently to be able to run in simulate tick mode. The
122  'intrusions points' are detailed below. The typical uses cases for using
123  simulated time are:
124 
125 @htmlonly
126 <pre>
127 
128  o Testing. A simulated system tick allows the test harness to have
129  deterministic control of the application's time. In addition, time
130  can be singled stepped, paused, accelerated (i.e. run N days in
131  M minutes).
132 
133  o Model in the Loop (MIL). An external application such as Matlab/
134  Simulink can be used to provide the simulated tick input as well
135  as capture the application's output and provide the application
136  inputs, i.e. connecting the application to a simulated physical
137  world. Note: 'connecting' a MIL application to a simulation such
138  as Matlib/Simulink requires a communication layer that is NOT
139  provided by Colony as well as specialization to Matlib/Simulink
140  that mates up with the afore mentioned communication layer. This
141  is left as an exercise to the reader :).
142 
143 
144  NOTES:
145  ------
146  o All timing sources, e.g. Cpl::Api::sleep(), provided by Colony.Core
147  have support for simulated ticks build in.
148 
149  o All of the Colony.Core's ITC message queues/mailbox servers, etc. also
150  have support for simulated ticks build in.
151 
152  o With respect to threads that are using simulated time, There needs to
153  be at least one thread (at any given time) that is a 'timing' source,
154  e.g. waiting on sleep(), timedWait(), etc. If your application uses
155  a Cpl::System::EventLoop (or a sub-class) instance - then this
156  requirement has been met. Worse case you a create a thread that
157  forever loops on sleep().
158 
159  o This interface is independent of the underlaying platform, i.e works
160  on ALL platforms.
161 
162  o NOT all threads and/or parts of an application need or should use
163  simulated time. For example the test harness (or communication
164  interface to a simulation tool) that provide the stimulus to
165  advance the simulated tick need to run in real-time. It is the
166  Application responsibility to properly selected what is and is
167  not using simulated time.
168 
169  - The platform's native/main thread ALWAYS runs in 'real-time', i.e.
170  call to SimTick::wait() never blocks. Also any Api::sleep()
171  calls from the native/main thread are redirected to the
172  Api::sleepInRealTime().
173 
174  o THREADING: This interface provides ONLY a functional simulation of
175  system ticks. These means the actual runtime timing of threads and/or
176  sequence of execution will NOT be the same as the 'live'
177  non-simulated application.
178 
179  - When using simulated ticks with multi-threaded applications, the
180  timing of the threads using the simulated ticks behavior is 'better'
181  if the thread supplying the ticks has LOWER priority than the
182  threads using the simulate ticks.
183 
184  - BEWARE of multi-core targets. Because many threads can actually
185  execute at the SAME time on multi-core boxes, the timing between
186  the tick source thread and the simulated-tick threads and between
187  simulated-tick threads is UNPREDICTABLE. The tick sequencing
188  is guaranteed, but the order within a tick OR timing relations to
189  the tick source OR inter-tick timing is NOT guaranteed.
190 
191  - Not all use cases using a simulated tick and threading will work
192  and/or are supported. The simulated tick logic only works for
193  event driven and/or frequency based where all actions for a time
194  period as completed in on pass of the "main loop". Other issues:
195 
196  o The assumption is that all calls to a mutex will cause the
197  thread to block across simulated ticks, i.e. mutex can ALWAYS
198  be successfully acquired and release within the context of
199  single simulated tick
200 
201  - Be careful about deleting threads when using Simulated Ticks -
202  brute force terminating threads in this scenario has unpredictable
203  and/or improper behavior. Always have threads run to completion
204  when using Simulate Ticks.
205 
206  o The DEFAULT implementation of this interface assumes that one
207  system tick EQUALS one millisecond.
208 
209 
210  HOW TO ADD SIMULATED TICK SUPPORT
211  ---------------------------------
212  o Enabling Simulated time is done at compile done. The following steps
213  are required:
214  1) Compile with the preprocessor symbol USE_CPL_SYSTEM_SIM_TICK
215  defined.
216 
217  2) Compile & Link the src/Cpl/System/_simtick directory. Also do NOT
218  compile/link the src/Cpl/System/xxx/_realtime directories
219 
220  o Typically thats all that needs to be done. The only additional work is if
221  application creates thread(s) that have an application specific (i.e. not
222  a Colony.Core Event Loop, mailbox server, etc.) 'forever' loop AND it
223  is desired for the thread(s) to use 'simulated time'. For this scenario
224  the application must add a call to this interface at the top of the
225  forever loop. See the example below:
226 
227  void myThreadEntryFunction( void* arg )
228  {
229  ...
230 
231  while(1)
232  {
233  // Block the main loop until the next simulated system tick
234  CPL_SYSTEM_SIM_TICK_TOP_LEVEL_WAIT();
235  ...
236  }
237  }
238  }
239
