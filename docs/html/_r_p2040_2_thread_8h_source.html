36/** This concrete class implements a Thread object that maps 1 to 1 with
37 a processor core on the Raspberry PI RP2040 micro-controller.
38
39 At least one thread must be created and at most only two threads can be
40 created. The first thread created executes on core0. The second thread
41 created executes on core1.
42
43 The first thread MUST be created before enableScheduling() is called. The
44 second thread that executes on core1 can be created or after enableScheduling()
45 is called. The core1 thread can also be destroyed and recreated after
46 enableScheduling() is called.
47
48 The name of the threads are hard coded to 'CORE0' and 'CORE1'.
49
50 The 'native' thread handle is the core number (i.e. 0 or 1).
51
52 Threads can ONLY be created using the Cpl::System::Thread::create() method.
53 When created threads:
54 - Only the 'runnable' and 'name' should be specified.
55 - The stack sizes are defined by the build symbols: PICO_STACK_SIZE
56 and PICO_CORE1_STACK_SIZE, e.g. -DPICO_STACK_SIZE=4096 -DPICO_CORE1_STACK_SIZE=2048
57 - Thread priority has NO meaning since both threads/cores execute concurrently.
58 - The thread(s) begin executing after enableScheduling() is called.
59
60 NOTES:
61 1. The threads do NOT 'execute' (i.e. call their Runnable object's run()
62 method till enableScheduling() is called. In addition the enableScheduling()
63 method NEVER returns.
64 2. The suspendScheduling() semantics suspends ALL processing on the 'other'
65 core. This includes interrupt processing on the other core.
66
67 */
