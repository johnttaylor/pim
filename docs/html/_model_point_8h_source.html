31/** This mostly abstract class defines the interface for a Model Point. A
32 Model Point contains an instance of a Point's data and is responsible for
33 managing thread safe access to the 'Point'. The intent of a Model point is
34 that it is the Application's canonical source for a 'Point'
35
36 NOTES:
37 1) The concrete leaf classes are responsible for ensuring type
38 safety.
39 2) All methods are atomic, i.e. a protected by the Model Base's mutex.
40 This means that all methods will block if there is a call in-progress
41 to ANY Model Point in a given Model Data Base.
42 3) All methods in this class ARE thread Safe unless explicitly
43 documented otherwise.
44 */

52 eLOCK, //!< Request to lock the MP. If the MP is already lock - the request is ignored and the update operation silent fails
53 eUNLOCK, //!< Request to unlock the MP. If the MP is already unlocked - the request is ignored and the update operation is completed
54 };

307 /** This method is used to attach a subscriber to a Model Point. Once
308 attached the Subscriber will receive a change notification (aka a
309 callback) every time the Model Point's data/state changes. Once, a
310 Subscriber is attached - it will stay attached to the application
311 calls detach().
312
313 There is no limit to the number of Subscribers that can attach to
314 a Model Point.
315
316 The attach() method can be called even if the Subscriber is already
317 attached. When this happens, the attach process is 'restarted', i.e.
318 the 'initialSeqNumber' is used for the Subscriber's sequence number.
319
320 The change-detect mechanism uses a sequence number. Each Model Point
321 and each Subscriber has sequence number. When the Subscriber's sequence
322 number does not equals the Model Point's sequence number - the Subscriber
323 receives a change notification and the Subscriber's sequence number is
324 updated to match the Model Point's sequence number at the time of
325 change notification. When a Subscriber attaches to Model Point with
326 an the 'seqNumber' argument set to SEQUENCE_NUMBER_UNKNOWN, the
327 Subscriber will get an 'immediate' change notification.
328
329 The callbacks for the Change Notifications are called as part of the
330 Data Model's Mailbox server. As part of the asynchronous processing (timers,
331 ITC, EventFlags, etc.) of the Data Model Mailbox server will also process
332 all pending Change Notification and invoke the call backs. What does
333 that all mean? The Change notifications are "local" to the Subscriber's
334 thread very similar to how the Cpl::Timers work. It also means that
335 no change notification callback will be called till the Mailbox server
336 loops back to the "top" of its forever loop.
337
338 NOTE: The Change Notification mechanism does NOT guarantee that a
339 client will receive a notification for EVERY change to a Model
340 Point (i.e. 'fast edges' could/will be missed). What is guaranteed
341 is that once a Model Point's value has 'settle' all registered
342 subscribers will have received change notification for the Model
343 Point's current value.
344 */
345 virtual void attach( SubscriberApi& observer,
346 uint16_t initialSeqNumber=SEQUENCE_NUMBER_UNKNOWN ) noexcept = 0;
347
348 /** This method is used to detach a Subscriber from a Model Point. See the
349 attach() method for more details about the Subscription/Change
350 Notification mechanism.
351
352 The detach() method can be called even if the Subscriber is NOT
353 currently attached. The detach() method can be called within the
354 Change Notification callback.
355 */
356 virtual void detach( SubscriberApi& observer ) noexcept = 0;
